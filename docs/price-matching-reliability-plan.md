# Price matching reliability plan

## 1. Purpose

`update-prices.js`（および定期実行している `update-prices-patched.js`）は、レンズの
新品・中古価格を kakaku.com から取得して `public/lens_data.json` の `price_info` に
書き込む。しかし現状、**別世代・別レンズの価格を取り違える** 不具合があり、
ユーザーに誤った価格を表示している。

この docs は、価格履歴機能（買い時表示）を作る前提として、価格マッチングを
信頼できる状態にするための設計を記録する。

## 2. 確認された不具合（実データ）

`price_info.kakaku_url` を複数レンズが共有している箇所を検出したところ、
27グループ・約60本で取り違えが起きていた。深刻度は3段階。

### レベル1: 表記ゆれによる重複行（価格は正しいが行が重複）
同一レンズが `Nikkor Z` / `NIKKOR Z` のように2行存在し、同じURL・同じ価格を持つ。
価格自体は正しいが、DBに重複レコードがあること自体が別の問題。
例: `Nikkor Z 35mm f/1.4` と `NIKKOR Z 35mm f/1.4`

### レベル2: 世代の取り違え（実害・大）
無印とII型/新型が同じkakaku URLを共有し、価格がほぼ同一（1円違い）。
初代に新型の価格が入っている（またはその逆）。
例:
- `FE 85mm F1.4 GM` と `FE 85mm F1.4 GM II` … 両方とも II型のURL (K0001649150)
- `FE 24-70mm F2.8 GM` と `GM II`
- `FE 70-200mm F2.8 GM OSS` と `OSS II`
- `FE 70-200mm F4 G OSS` / `Macro G OSS` / `Macro G OSS II`（3本同一URL）
- `XF16-55mmF2.8 R LM WR` と `WR II`
- `Samyang AF 50mm F1.4 FE` と `II FE`

### レベル3: 完全に別レンズの誤マッチ（実害・最大）
焦点距離もタイプも違う無関係なレンズが同じURLを共有。
例:
- `RF100-500mm` と `RF 500mm F4 L`
- `FE 28mm F2`（単焦点）と `FE 28-70mm F2 GM`（ズーム）
- `AF 35mm f/2.8 FE` と `FE 16-35mm F2.8 GM II`
- `AF 75mm F1.8 FE` と `FE 16mm F1.8 G`
- `RF 28mm F2.8 STM` と `RF28-70mm F2.8 IS STM`
- `Samyang AF 14mm F2.8 FE`（単焦点）と `AF 14-24mm F2.8 FE`（ズーム）

## 3. 根本原因

`scripts/update-prices.js` のロジック:

1. `scrapePrice()`: `price_info.kakaku_url` がキャッシュされていればそれを直接使う。
   → **思想は正しい（K番号を固定IDとして信頼）。**
2. キャッシュがなければ `searchVariants()` で kakaku を検索し、検索結果の
   **最初の `/item/K` リンク** を採用してキャッシュする。

不具合は2点:

- **`searchVariants()` の短縮ロジックが世代・グレード語を削る**（79行目付近）。
  検索失敗時に `GM` / `G` / `L` / `VR` / `OSS` などの修飾子や f値を削った
  短いクエリで再検索するため、「FE 85mm F1.4 GM」→「FE 85mm F1.4」となり
  新しいII型が上位ヒットして誤マッチする。単焦点の焦点距離だけが残って
  ズームと衝突するケースも同様。
- **誤キャッシュが自己強化される**。一度誤ったK番号がキャッシュされると、
  以降はキャッシュ分岐でそのまま使われ、価格が永久に間違い続ける。
  検出する仕組みもない。

## 4. 設計方針

骨格（K番号を固定IDとして直接アクセスする）は維持し、「URL確定を信頼できる
ものにする」ことに集中する。ASIN直リンク化で確立した「人手で正しいIDを確定 →
以降そのIDを信頼」パターンを価格側にも適用する。

### 4.1 kakaku ID マップの導入（ASIN方式の踏襲）
- `data/kakaku_id_map.json` … `{ "レンズ名": "K0001649150" }` の対応表をコミット資産にする。
- `scrapePrice()` の優先順位を変更:
  1. `kakaku_id_map` にエントリがあれば、そのK番号のURLを直接使う（検索しない）
  2. なければ従来の `price_info.kakaku_url` キャッシュを使う
  3. それもなければ検索（ただし 4.2 の安全策付き）
- これにより、人手で確定したレンズは二度とズレない。

### 4.2 検索フォールバックの安全化
- `searchVariants()` から **世代・グレードを示す語（GM/G/L/II/無印区別/VR/OSS など）を
  削る短縮を禁止**する。f値や焦点距離も保持する。短縮するのは末尾の
  マウント表記・色名・冗長な空白程度に留める。
- 検索でK番号を得たら、**確定前に検証**する: 取得ページの商品名に、レンズ名の
  主要トークン（焦点距離・F値・世代語）が含まれるか緩く照合し、外れていれば
  採用せず「要手動確認」としてログに出す（誤キャッシュを防ぐ）。

### 4.3 取り違え検出を常設（--audit）
- `node scripts/update-prices.js --audit` で、複数レンズが同じ `kakaku_url` を
  共有している箇所を一覧出力する（section 2 の検出を機能化）。
- 表記ゆれ重複（レベル1）と実害（レベル2/3）を区別して表示する。
- 定期実行のたびに、または更新後に audit を回せば、誤マッチに即座に気づける。

### 4.4 既存の誤データのクリーンアップ
- まず `--audit` で全件洗い出し。
- レベル2/3の各レンズについて、正しい kakaku 製品ページ（K番号）を人手で確定し
  `data/kakaku_id_map.json` に記入。
- マップ適用 → 該当レンズのみ価格再取得 → `--audit` で解消を確認。
- レベル1（表記ゆれ重複行）は価格の問題ではなくDB重複の問題。別タスクとして、
  重複レコードの統合方針を検討する（今回のスコープ外）。

## 5. スコープと非スコープ

### このタスクでやること
- `data/kakaku_id_map.json` の導入と、`scrapePrice()` の参照順序変更
- `searchVariants()` の世代・グレード語温存への修正
- 検索結果の確定前検証
- `--audit` モードの追加
- 誤データ修正用の worklist 生成（ASIN方式と同様の CSV）

### やらないこと（非スコープ）
- 価格履歴の蓄積機能（買い時表示）… これはマッチング健全化の後の別タスク
- 表記ゆれによる重複レコードの統合 … DB構造の別タスク
- `lens_data.json` のフォーマット全体変更（最小diffを維持）
- 推薦ロジック / API / localStorage / UI への変更

## 6. 実装後の検証

1. `--audit` がレベル2/3グループを0件（またはマップ確定済みのみ）にできること
2. `kakaku_id_map` に登録したレンズは、再取得しても正しいK番号を使い続けること
3. `searchVariants()` が「FE 85mm F1.4 GM」で II型URLを拾わないこと
   （世代語を保持した検索になっていること）
4. `npm run build` / `node scripts/check_lens_data.js` が通ること
5. 価格更新の差分が、対象レンズの `price_info` のみであること（最小diff）

## 7. 補足: なぜ履歴機能の前にこれをやるか

価格履歴（買い時表示）は、日々の価格を時系列で蓄積して「過去最安」「値下がり」を
見せる機能。土台の現在価格が誤っていると、誤った履歴が延々と蓄積され、誤った
買い時シグナルを出してしまう。マッチングを健全化してから履歴を積み始めるのが正しい順序。
