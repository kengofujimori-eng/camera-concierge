## 1. プロジェクト概要（確定事項）

camera-concierge は、日本語のレンズ推奨 Web アプリ。

- スタック: Next.js 14、ローカル Mac 開発、GitHub（`kengofujimori-eng/camera-concierge`）
- 基本コンセプト: チャットボット的な対話から、ユーザーに最適なレンズを導き出す。
- データ保存: ブラウザの localStorage（端末・ブラウザをまたぐと保持されない）。
  当面 DB は導入せず、ローカル保存で進める方針。

### 作業主体（現状の理解）

- 本筋・メイン実装: ChatGPT + Codex
- 俯瞰的視点からの構成・改善: Claude + Claude Code
- 全体の交通整理・最終判断: 人間（プロジェクトオーナー）

> 注: 上記は現状の役割理解であり、固定的な分担ではない。分業の仕方は協議で調整する。

## 2. 現在の画面構成（確定事項）

### `/warehouse`（レンズ倉庫・本番）

- 所有 / 欲しい レンズの管理（localStorage 保存）
- 焦点距離カバレッジ（横棒グラフによる所有・欲しいレンズの焦点距離可視化）
- レンズの手動追加・削除
- 1レンズあたりの濃い情報を、折りたたみ（accordion）で収納:
  - AIの分析（長所 / 短所 / 結論）
  - 新品 / 中古 価格
  - 使いこなしレビュー（総括「一言でいうと」+ 強み要点 / 注意要点 +
    解像とF値・ボケ・収差・周辺減光逆光・比較・サイズ重量・根拠 等の細分化項目）
  - 新品で買う（Amazon / 楽天 / Yahoo）
  - 中古で買う（キタムラ）
  - 作例・レビュー（Photo Yodobashi / Google 作例検索 / 海外レビュー）

### `/warehouse-gallery`（武器庫ギャラリー・実験ページ）

- `/warehouse` とは独立した実験ページ。現在 v9 系で開発中。
- 「データベース感のある一覧」から「レンズを愛でるコレクション体験」への
  ビジュアル刷新プロトタイプ。
- 世界観: マットブラック / オニキス、横一列のフロストグラスカード、
  モノクロ基調 + 一点挿し、静謐さ重視、台帳のように詰め込まない。
- 二段階表示: 第一段階＝静かな概要、第二段階＝下からせり上がる詳細パネル。
- 現状はデモ専用のハードコードデータ（`LENSES` 配列）で動作。本番データには未接続。
- 詳細パネルは文字タブ化を進行中（AI解析 / 価格・購入 / レビュー・作例）。

### 現状の課題認識

- `/warehouse` と `/warehouse-gallery` は「レンズを見る」という同一目的を、
  異なる UI で二重に実装している状態。
- 両者の実質的な差は「焦点距離カバレッジの有無」と「情報量・収納方式」にほぼ集約される。
- 倉庫の折りたたみ式は、全項目展開時に不要な部分まで開き、可読性が下がる傾向がある。
- ギャラリーのタブ式は、文字情報を集中的に見せられ、可読性で優位の可能性がある。

## 3. 協議中の提案: ギャラリーへの統合

> 以下は提案であり、未確定。合意形成の対象。

### 提案の骨子

最終的に `/warehouse`（倉庫）の機能を `/warehouse-gallery`（ギャラリー）側へ統合し、
ギャラリーの世界観を持つ単一の画面に一本化する。

理由（提案根拠）:
- 同一目的の画面を二重に保守するコストを解消できる。
- UI の理想形（タブによる集中的な情報表示、静謐な世界観）はギャラリー側にある。
- ユーザーにとって「どちらを見るか」の迷いがなくなる。

統合後にギャラリーが備えるべき機能:
- 所有 / 欲しい の切り替え（倉庫の状態管理を移植）
- 焦点距離カバレッジ（倉庫から移植、世界観に合わせて再設計）
- レンズごとの濃い情報を、タブ + 折りたたみで整理して収納

### 役割分担についての論点（要協議）

- 倉庫（`/warehouse`）の状態管理・localStorage ロジックは、現状メイン実装側の領域。
- 統合はこの領域に踏み込むため、ギャラリー側だけで一方的に進めると衝突リスクが高い。
- 「どちらの画面を正とするか」「統合の主担当は誰か」を協議で決める必要がある。

## 4. 協議中の提案: 統合の段階計画

> 以下は提案であり、未確定。順序・粒度は協議で調整する。

一度に統合すると事故リスクが高いため、段階に分割する案:

**第1段階 — 詳細パネルの器を完成させる**
ギャラリーの詳細パネルを、倉庫レベルの情報量を綺麗に収納できる器にする
（タブ + 必要に応じた折りたたみ）。倉庫の機能ロジックには触れないため、
統合の可否に関わらず着手でき、無駄にならない。現在進行中の v9 系がこれに該当。

**第2段階 — 所有 / 欲しい の状態をギャラリーへ**
倉庫が持つ「所有 / 欲しい」の状態概念をギャラリーに導入。
localStorage の保存キー・形式を 1 つに統一することを、この段階の前に設計事項として決める。

**第3段階 — 焦点距離カバレッジの移植**
倉庫の焦点距離カバレッジを、ギャラリーの世界観に合わせて再設計し移植する。

**第4段階 — データ一本化と本番昇格**
ハードコードの `LENSES` を本番データに接続。ギャラリーを実験ルートから本番へ昇格させ、
旧倉庫を引退させる。

### 段階ごとの越境の有無

- 第1段階: ギャラリー領域内で完結（越境なし）。
- 第2段階以降: 倉庫の実装・localStorage ロジックに踏み込むため、越境あり。要事前合意。

## 5. 協議中の提案: 詳細パネルの情報設計

> 以下は提案であり、未確定。

倉庫の濃い情報をギャラリーの詳細パネルへどう写像するかについて、2つの方向性がある。

**案A — 総括 + 導線型**
ギャラリーは各情報の「総括」のみ静かに見せ、細分化された深い評価
（解像・ボケ・収差・周辺減光 等）は別途参照させる導線を置く。
- 利点: 静謐さを保ちやすい。情報の二重保守を避けられる。
- 前提: 「深い情報の正（マスター）はどこか」を1箇所に定める必要がある。

**案B — 畳んで全部持つ型**
ギャラリーにも細分化項目を、初期状態は畳んだ形で全て持たせ、クリックで展開する。
- 利点: 1画面で情報が完結する。
- 注意: 統合が前提なら「二重保守」の問題は消えるが、展開項目が増えると
  静謐さが薄れるリスクがある。

> 統合（第3節）を前提とする場合、案Bの「二重保守」懸念は解消される
> （倉庫が引退するため、情報源が1つになる）。よって統合方針の合意内容によって、
> この情報設計の最適解も変わる。両者は連動して決めるべき論点。

## 8. 共同作業ルール

この章は、ChatGPT / Codex / Claude / Claude Code が同じ前提で作業するための正式な共同作業ルールである。

### Source of truth

- `origin/main` を唯一の正とする。
- 各作業主体は、作業開始時に `origin/main` の最新状態を確認する。
- ローカルの diverged branch、実験 branch、各AIの一時作業状態を正として扱わない。
- 方針・構造・統合判断に関わる内容は、最終的にこの `docs/ARCHITECTURE.md` へ反映してから実装へ進む。

### Route status

- `/warehouse` は現行本番のレンズ倉庫である。
- `/warehouse-gallery` は統合候補の実験領域である。
- `/warehouse-gallery` は、本番倉庫の置き換え候補ではあるが、現時点で本番 route ではない。
- ユーザー導線、保存形式、実データ接続に関する正は、合意があるまで `/warehouse` 側にある。

### Boundary changes

以下は越境変更として扱う。単独判断で進めず、事前に本書または専用 docs で方針を明文化する。

- localStorage key / schema / migration。
- `/warehouse` の本番UI、保存、削除、一覧、詳細表示。
- 本番データ構造、`public/lens_data.json`、価格・購入リンク・画像・review link。
- `/warehouse` と `/warehouse-gallery` の route 統合、redirect、導線変更。
- 保存済みレンズ、AI分析、使いこなしレビュー、Deep Review 表示の情報源変更。

### Role ownership

- Claude / Claude Code は、当面 `/warehouse-gallery` の UI・体験・情報設計に限定して作業する。
- Claude / Claude Code は、localStorage、本番 `/warehouse`、本番データ構造、route 統合には直接踏み込まない。
- ChatGPT / Codex は、本番挙動、データ、localStorage、migration、統合計画の主担当とする。
- 詳細パネルの情報写像は、Claude / ChatGPT / Codex の共同作業ゾーンとする。
  - Claude / Claude Code: 体験、見せ方、視線誘導、情報の収め方。
  - ChatGPT / Codex: 本番データ、保存形式、既存UIとの差分、移行可能性。
  - 人間: 最終判断と優先順位。

### LocalStorage inventory

- localStorage 棚卸しは Codex が行う。
- Codex は、保存 key、保存値の形、読み込み箇所、書き込み箇所、migration リスクを docs に記録する。
- Claude / Claude Code は、その棚卸し結果を参照してから UI / 体験設計を進める。
- 棚卸し前に、Claude / Claude Code が保存形式や統合方針を仮定して実装しない。

### Architecture document ownership

- `docs/ARCHITECTURE.md` の正式更新は ChatGPT 主導、人間承認とする。
- Claude / Claude Code からの補足や提案は歓迎するが、正式な現在地として反映する前に人間が承認する。
- Codex は、実装で判明した事実、棚卸し結果、差分リスクを docs に反映する。
- 既存章の大幅な書き換えは、追記よりも慎重に扱う。原則として、まず追加章または補足節で現在地を明確にする。

## 9. レンズ倉庫大改修の判断基準

この章は、`/warehouse` と `/warehouse-gallery` の統合や、本番レンズ倉庫の大きな改修を進めるかどうかの判断基準である。

### 大改修を行う条件

以下を満たす場合、レンズ倉庫の大改修または統合を検討してよい。

- `/warehouse-gallery` の体験が、現行 `/warehouse` より明確に見やすく、保存済みレンズの再確認に向いている。
- 詳細パネルが、現行 `/warehouse` の AI分析、価格、購入リンク、レビュー・作例、使いこなし情報を無理なく収納できる。
- localStorage 棚卸しが完了し、保存 key / schema / migration リスクが docs 化されている。
- 本番データ構造と `/warehouse-gallery` の表示モデルの対応関係が明文化されている。
- route 統合時の rollback 方針がある。
- 既存の warehouse 保存済みデータを破壊しない方針がある。
- E2E selector、既存 localStorage、購入リンク、affiliate 表示、価格表示を保護できる。
- 統合作業の主担当とレビュー担当が明確である。

### 大改修を急がない条件

以下の場合は、統合や本番置き換えを急がない。

- `/warehouse-gallery` が demo data / hardcoded data 前提のまま。
- localStorage の保存形式、読み書き箇所、既存ユーザー影響が棚卸しされていない。
- 詳細パネルの情報写像が、見た目だけで決まり、本番データとの対応が未確認。
- `/warehouse` の機能をどこまで維持するか未確定。
- 焦点距離カバレッジ、所有 / 欲しい、削除、保存、購入リンク、レビュー導線の移植方針が曖昧。
- 公開βの実利用で、倉庫より AI相談品質や推薦品質の課題が優先されている。
- 画面の好みだけで大改修を進めようとしている。
- API / Dify / lens data / localStorage / route 統合を同時に変更しようとしている。

### 統合判断の合格ライン

`/warehouse-gallery` を本番統合候補として扱うには、最低限以下を満たす必要がある。

- 現行 `/warehouse` の主要機能を失わない。
  - 保存済みレンズ表示。
  - 所有 / 欲しいの区別。
  - 手動追加 / 削除。
  - 価格・購入リンク。
  - AI分析。
  - 使いこなしレビュー。
  - レビュー・作例リンク。
  - 焦点距離カバレッジ。
- 既存 localStorage データが維持される、または安全な migration が定義されている。
- 実データ接続時に、demo data 前提の見せ方が破綻しない。
- モバイルで主要操作が成立する。
- 既存 E2E hooks / data-testid を壊さない。
- 旧 `/warehouse` へ戻せる rollback 方針がある。
- 統合後の route 方針（`/warehouse` を維持するか、gallery を昇格するか）が明確である。

### 推奨する進め方

1. `/warehouse-gallery` では、まず UI・体験・詳細パネルの器を磨く。
2. Codex が `/warehouse` の localStorage、保存形式、データ参照、E2E selector を棚卸しし、docs に記録する。
3. Claude / Claude Code は、棚卸し結果を参照し、詳細パネルへの情報写像を提案する。
4. ChatGPT / Codex は、本番データ構造、migration、route 統合、rollback 方針を整理する。
5. 人間が、統合する / しない / 先送りする、を判断する。
6. 統合する場合も、一度に route を置き換えず、まず実データ接続の pilot を行う。
7. pilot で保存済みレンズ、価格、購入リンク、レビュー導線、モバイル表示を確認する。
8. 問題がなければ、PR単位で小さく本番昇格を検討する。

### Guardrails

- 統合は「見た目が良い」だけでは進めない。
- localStorage migration を暗黙に行わない。
- `/warehouse` を壊して `/warehouse-gallery` を試さない。
- API / Dify / recommendation logic / lens data 更新と倉庫統合を同時に行わない。
- Claude / Claude Code の実験成果は尊重するが、本番統合は ChatGPT / Codex の移行計画と人間承認を経る。
