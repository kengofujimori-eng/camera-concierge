# Price history and buy-timing plan

## 1. Purpose

レンズの価格は日々変動する。現状 `public/lens_data.json` の `price_info` は
「最新の1点」しか持たないため、ユーザーは「今が買い時か」を判断できない。

このドキュメントは、価格を時系列で蓄積し、欲しいリスト/倉庫で「過去最安」
「値動き」「買い時シグナル」を見せる機能（②）の設計を記録する。これは
camera-concierge のコア体験——倉庫に入れた欲しいレンズの購入計画を立てる——
の中核になる。

前提として、価格マッチングの信頼性確保（price-matching-reliability-plan.md）は
完了済み。クリーンな現在価格の上に履歴を積み始める。

## 2. 全体構成（3部品）

| 部品 | 役割 | 本ドキュメントのスコープ |
| --- | --- | --- |
| 1. データ層 | 毎日の価格を時系列で蓄積する | **今回実装** |
| 2. ロジック層 | 履歴から指標（最安・値動き・買い時）を計算 | 設計のみ記載・実装は次段階 |
| 3. UI層 | 倉庫/欲しいリストでスパークライン・買い時バッジ表示 | 設計のみ記載・実装は次段階 |

履歴は今日から蓄積を開始する（過去コミットからの遡及復元はしない。
取り違え時代のデータが混在し品質が不揃いなため、クリーンな起点を優先）。

## 3. 部品1: データ層（今回実装）

### 3.1 保存先
- 新規ファイル `public/price_history.json` に分離する。
  （lens_data.json には履歴を入れない。肥大化と巨大 diff を避けるため）

### 3.2 構造
```json
{
  "FE 85mm F1.4 GM": [
    { "date": "2026-06-16", "new": 173607, "used": 100200 },
    { "date": "2026-06-17", "new": 172000, "used": 99800 }
  ],
  "FE 85mm F1.4 GM II": [
    { "date": "2026-06-16", "new": 226640, "used": 204100 }
  ]
}
```
- レンズ名（lens_data.json の `name` と一致）をキー、日付昇順の配列。
- 各要素は `date`(YYYY-MM-DD) / `new` / `used` のみ。
  価格が取得できなかった場合は null を入れる（その日「取れなかった」記録も価値がある）。
- 画像URL・kakaku_url は履歴に含めない（最新は lens_data.json にある）。

### 3.3 蓄積タイミングと重複処理
- `update-prices.js` が価格を取得し lens_data.json を更新するのと同時に、
  price_history.json にもその日の `{date, new, used}` を記録する。
- **同一 date のエントリが既にあれば上書き、なければ追記**する。
  （同じ日に複数回 --force-all を走らせても 1日1点に保つ）
- レンズ名が price_history.json に未登録なら新しい配列を作る。
- lens_data.json から削除されたレンズの履歴は、当面は残す（無害）。
  将来クリーンアップする場合は別タスク。

### 3.4 実装方針
- update-prices.js の保存処理（lens_data.json 書き込み箇所）に、
  price_history.json への追記を並行して行う処理を追加する。
- price_history.json が存在しない場合は `{}` から作る。
- 中間保存（SAVE_INTERVAL ごと）のタイミングでも履歴を書き出し、
  途中で止まってもその時点までの履歴が残るようにする。
- JSON.stringify での整形は許容（price_history.json は機械生成専用ファイルで、
  人間が手編集しないため、最小 diff 制約は適用しない）。

## 4. 部品2: ロジック層（設計のみ・次段階）

履歴から以下を計算する関数群を用意する想定（実装は次タスク）。

- **過去最安**: 記録された new/used の最小値とその日付。
- **現在が最安か**: 最新値が過去最安と一致するか。
- **前回比**: 直近2点の差分（値上がり/値下がり/横ばい）。
- **買い時シグナル**（初期はシンプルに）:
  - 「記録中の最安値（またはそれに近い、例: 最安+2%以内）」→ 買い時
  - データ点が少ない初期は「データ蓄積中」と表示し、誤ったシグナルを出さない
- 高度化（3ヶ月最安、移動平均比など）は履歴が十分溜まってから。

## 5. 部品3: UI層（設計のみ・次段階）

倉庫（warehouse）と欲しいリストで以下を見せる想定（実装は次タスク）。

- **価格スパークライン**: 各レンズカードに小さな推移グラフ。
- **買い時バッジ**: 「過去最安」「値下がり中」などのラベル。
- **購入計画への接続**: 欲しいレンズに目標額を設定 → 現在価格との差・
  買い時シグナルと組み合わせて「あと○円」「今が好機」を表示。
  （ロードマップの「貯金トラッカー」体験につながる）

## 6. スコープと非スコープ

### 今回やること（部品1）
- `public/price_history.json` の新規作成（初期 `{}`）
- `update-prices.js` への履歴追記処理の追加（同日上書き・追記マージ）
- 今日の価格更新実行で最初の1点が記録されることの確認

### やらないこと（非スコープ）
- 過去コミットからの履歴遡及復元
- ロジック層・UI層の実装（部品2・3は次段階）
- lens_data.json の構造変更
- 推薦ロジック / API / 既存UI への変更

## 7. 確認

1. update-prices.js 実行後、public/price_history.json が生成され、
   各レンズに当日の {date, new, used} が記録されること
2. 同日に再実行しても、その日のエントリが重複せず上書きされること
3. lens_data.json の既存挙動（最新価格の更新）が変わらないこと
4. npm run build / db:check が通ること
